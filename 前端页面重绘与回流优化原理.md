# Node.js 内存泄露排查与诊断指南

## 引言
Node.js 服务在长期运行后，如果 RSS 持续增长、GC 频率升高、响应延迟抖动明显，往往意味着存在内存泄露。对于需要处理 [高并发处理](https://main-ayx-app.com.cn) 的线上服务，这类问题会迅速放大，最终演变为 OOM、进程重启或性能雪崩。因此，建立一套可重复的排查与诊断方法非常必要。

## 核心原理分析
Node.js 的内存问题通常分为两类：一类是“真正泄露”，即对象被无意持有，导致 V8 无法回收；另一类是“高水位增长”，例如缓存、队列、缓冲区使用不当，看似增长但其实是业务设计使然。诊断时重点关注三项指标：`heapUsed`、`heapTotal`、`rss`。其中 `heapUsed` 持续上升且 GC 后不回落，通常更接近泄露；若 `rss` 上升但 `heapUsed` 稳定，则要检查原生内存、Buffer、外部库或线程池资源。

常见根因包括：全局变量误用、Map/Set 无上限增长、事件监听器未解绑、闭包捕获大对象、定时器未清理，以及请求上下文被长链路引用。排查时建议结合 heap snapshot、allocation timeline 和 `--inspect`，比单纯看日志更有效。

## 代码示例
下面示例展示如何在服务中监控内存，并在异常增长时输出快照线索，便于定位问题：

```js
const v8 = require('v8');

function reportMemory() {
  const mem = process.memoryUsage();
  const heap = v8.getHeapStatistics();

  console.log({
    rssMB: (mem.rss / 1024 / 1024).toFixed(2),
    heapUsedMB: (mem.heapUsed / 1024 / 1024).toFixed(2),
    heapTotalMB: (mem.heapTotal / 1024 / 1024).toFixed(2),
    externalMB: (mem.external / 1024 / 1024).toFixed(2),
    heapLimitMB: (heap.heap_size_limit / 1024 / 1024).toFixed(2)
  });
}

setInterval(reportMemory, 10000);

// 典型修复：清理定时器与监听器
function startJob(emitter) {
  const timer = setInterval(() => {
    // do work
  }, 1000);

  const onData = () => {};
  emitter.on('data', onData);

  return () => {
    clearInterval(timer);
    emitter.off('data', onData);
  };
}
```

## 总结
Node.js 内存泄露排查的关键，不是“猜哪里有问题”，而是通过指标、快照和对象引用链逐步缩小范围。先确认是 V8 堆泄露还是外部内存增长，再结合监听器、缓存、闭包和定时器逐项排除，通常能快速定位根因。对于线上服务，建议把内存监控、阈值告警和快照采集做成标准流程，避免问题积累到不可控阶段。
